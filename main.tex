\section{Definición del lenguaje: MiniPeka}

MiniPeka es un lenguaje de programación simple diseñado con fines educativos
para ilustrar el proceso completo de compilación. Su sintaxis está inspirada
en lenguajes imperativos como C y Python, simplificada para facilitar la
implementación de cada fase del compilador. MiniPeka soporta declaración de
variables, operaciones aritméticas, estructuras de control básicas, definición
de funciones y entrada/salida por consola.

\subsection{Características principales}

\begin{itemize}
    \item \textbf{Tipado estático:} todas las variables deben declararse con
          su tipo antes de usarse. Los tipos disponibles son \texttt{int},
          \texttt{float} y \texttt{bool}.

    \item \textbf{Funciones con parámetros:} el lenguaje permite definir
          funciones con parámetros tipados y valor de retorno explícito.

    \item \textbf{Estructuras de control:} soporta \texttt{if}/\texttt{else}
          y el ciclo \texttt{while}.

    \item \textbf{Operadores aritméticos y relacionales:} suma, resta,
          multiplicación, división, y comparadores como \texttt{==},
          \texttt{!=}, \texttt{<}, \texttt{>}.

    \item \textbf{Instrucción de salida:} la función \texttt{print()}
          permite mostrar valores en consola.

    \item \textbf{Comentarios:} se soportan comentarios de una línea con
          el símbolo \texttt{\#}.
\end{itemize}

\subsection{Ejemplo representativo del lenguaje}

El siguiente fragmento muestra las capacidades principales de MiniPeka:

\begin{lstlisting}[caption={Ejemplo general de MiniPeka}]
# Declaracion de variables
int x = 5;
int y = 3;
float resultado = 0.0;

# Funcion que suma dos enteros
fun suma(int a, int b) : int {
    return a + b;
}

# Uso de la funcion y estructura de control
int total = suma(x, y);

if (total > 6) {
    print(total);
} else {
    print(0);
}

# Ciclo while
int i = 0;
while (i < total) {
    i = i + 1;
}
\end{lstlisting}

\section{Gramática preliminar}

La gramática de MiniPeka está definida como una gramática libre de contexto
(CFG). Se utiliza la notación BNF extendida donde \texttt{|} indica
alternativas, los elementos entre corchetes \texttt{[ ]} son opcionales y
los elementos entre llaves \texttt{\{ \}} pueden repetirse cero o más veces.

\subsection{Producciones principales}

\section{Análisis léxico}

El analizador léxico (scanner) de MiniPeka es la primera fase del compilador.
Su responsabilidad es leer el archivo fuente con extensión \texttt{.peka}
carácter por carácter, agruparlos en unidades significativas llamadas
\textit{lexemas} y clasificarlos en categorías denominadas \textit{tokens}.
Adicionalmente, el scanner descarta los comentarios y espacios en blanco,
y reporta con precisión cualquier carácter inválido indicando la línea y
columna donde ocurrió.

\subsection{Implementación}

El scanner está implementado en Python en el archivo \texttt{lexer.py}.
Utiliza el módulo \texttt{re} de la biblioteca estándar para definir los
patrones de cada token mediante expresiones regulares. Todos los patrones
se compilan en un único patrón maestro usando grupos nombrados
(\texttt{(?P<nombre>regex)}), lo que permite identificar qué regla
coincidió en cada posición del código fuente.

La clase principal \texttt{Lexer} recibe el código fuente como cadena de
texto y expone el método \texttt{tokenizar()}, que recorre el código de
izquierda a derecha produciendo la secuencia de tokens. El punto de entrada
del programa acepta un archivo \texttt{.peka} como argumento desde la
terminal:

\begin{lstlisting}[language=bash, caption={Uso del scanner desde terminal}]
python lexer.py <archivo.peka>
\end{lstlisting}

\noindent
Las decisiones de diseño más importantes del scanner son:

\begin{itemize}
    \item \textbf{Prioridad de palabras reservadas:} las palabras reservadas
          se definen antes que el token \texttt{ID} en la lista de reglas,
          garantizando que nunca sean clasificadas como identificadores.

    \item \textbf{Prioridad de flotantes sobre enteros:} el patrón
          \texttt{NUM\_FLOAT} se define antes que \texttt{NUM\_INT} para
          que una cadena como \texttt{3.14} sea reconocida como número
          real y no como el entero \texttt{3} seguido de un punto suelto.

    \item \textbf{Prioridad de operadores dobles:} los operadores
          \texttt{==} y \texttt{!=} se definen antes que \texttt{=} para
          evitar que sean tokenizados como dos operadores separados.

    \item \textbf{Recuperación ante errores:} cuando se encuentra un
          carácter inválido, el scanner lo reporta con línea y columna
          exactas y continúa procesando el resto del archivo sin detenerse,
          permitiendo detectar todos los errores léxicos en una sola pasada.

    \item \textbf{Descarte de comentarios y espacios:} los comentarios y
          espacios en blanco son reconocidos pero no generan tokens,
          manteniendo limpia la secuencia de salida.
\end{itemize}

\section{Análisis sintáctico}

El analizador sintáctico (parser) de MiniPeka es la segunda fase del
compilador. Recibe la secuencia de tokens producida por el lexer y verifica
que su estructura corresponda a la gramática del lenguaje. Si la secuencia
es válida, construye el \textit{Árbol de Sintaxis Abstracta} (AST), que
es la representación estructural del programa utilizada por las fases
posteriores del compilador.

\subsection{Implementación}

El parser está implementado en Python en el archivo \texttt{parser.py}
como un \textbf{parser LL recursivo descendente}. En este enfoque existe
una función por cada regla de la gramática, y el parser decide qué
producción aplicar mirando el token actual sin necesidad de retroceder.

La clase principal \texttt{Parser} recibe la lista de tokens del lexer y
expone el método \texttt{parsear()}, que recorre los tokens de izquierda
a derecha construyendo el AST. Se ejecuta desde la terminal así:

\begin{lstlisting}[language=bash, caption={Uso del parser desde terminal}]
python parser.py <archivo.peka>
\end{lstlisting}

\noindent

una construcción del lenguaje:

\end{table}

\noindent
Las decisiones de diseño más importantes del parser son:

\begin{itemize}
    \item \textbf{Parser LL recursivo descendente:} cada regla de la
          gramática tiene su propia función en el parser. Es el enfoque
          más legible y sencillo de implementar para una gramática como
          la de MiniPeka.

    \item \textbf{Integración con el lexer:} el parser importa directamente
          la clase \texttt{Lexer} de \texttt{lexer.py}. Si el lexer detecta
          errores léxicos, el parser se detiene antes de intentar construir
          el AST, evitando errores en cascada.

    \item \textbf{Recuperación ante errores:} cuando el parser encuentra
          un token inesperado, reporta el error con la línea y el token
          encontrado, y continúa intentando procesar el resto del programa.

    \item \textbf{AST simplificado:} a diferencia del árbol de derivación,
          el AST no incluye los nodos intermedios de la gramática como
          \textit{expr}, \textit{term} o \textit{factor}. Solo conserva
          los nodos semánticamente relevantes.
\end{itemize}

\subsection{Árbol de derivación}

El árbol de derivación muestra cómo una cadena se genera a partir del
símbolo inicial de la gramática aplicando las producciones paso a paso.
A diferencia del AST, el árbol de derivación incluye todos los nodos
intermedios de la gramática. Para la expresión \texttt{int z = x + y;}
la derivación es:

\shorthandoff{<>}
\begin{align*}
\langle statement \rangle
    &\Rightarrow \langle decl \rangle \\
    &\Rightarrow \langle type \rangle\ \textbf{id}\ \texttt{=}
                 \ \langle expr \rangle\ \texttt{;} \\
    &\Rightarrow \texttt{int}\ \textbf{z}\ \texttt{=}
                 \ \langle expr \rangle\ \texttt{;} \\
    &\Rightarrow \texttt{int}\ \textbf{z}\ \texttt{=}
                 \ \langle expr \rangle\ \langle op \rangle
                 \ \langle term \rangle\ \texttt{;} \\
    &\Rightarrow \texttt{int}\ \textbf{z}\ \texttt{=}
                 \ \langle term \rangle\ \texttt{+}
                 \ \langle term \rangle\ \texttt{;} \\
    &\Rightarrow \texttt{int}\ \textbf{z}\ \texttt{=}
                 \ \textbf{x}\ \texttt{+}\ \langle term \rangle\ \texttt{;} \\
    &\Rightarrow \texttt{int}\ \textbf{z}\ \texttt{=}
                 \ \textbf{x}\ \texttt{+}\ \textbf{y}\ \texttt{;}
\end{align*}
\shorthandon{<>}

\noindent
El AST correspondiente simplifica este árbol eliminando los nodos
intermedios \texttt{decl}, \texttt{expr} y \texttt{term}, conservando
únicamente los nodos semánticamente relevantes:

\vspace{0.3cm}
\begin{center}
\shorthandoff{<>}
\begin{tikzpicture}[
    level distance=1.1cm,
    level 1/.style={sibling distance=4.0cm},
    level 2/.style={sibling distance=2.0cm},
    every node/.style={
        draw=blue!50, fill=blue!7,
        rounded corners=3pt,
        font=\small\bfseries,
        minimum width=1.8cm,
        minimum height=0.6cm,
        inner sep=3pt
    },
    term/.style={
        draw=teal!60!black, fill=teal!10,
        font=\small\ttfamily\bfseries
    },
    edge from parent/.style={
        draw=gray!60, semithick
    }
]
\node {Declaracion}
    child { node[term] {int} }
    child { node[term] {z} }
    child { node {Operacion +}
        child { node[term] {ID: x} }
        child { node[term] {ID: y} }
    };
\end{tikzpicture}
\shorthandon{<>}
\end{center}

\section{Análisis semántico}

El análisis semántico verifica que el \textit{significado} del programa
sea correcto más allá de su estructura sintáctica. El módulo
\texttt{semantico.py} recibe el AST generado por el parser y realiza
las siguientes tareas:

\begin{enumerate}
    \item Construir una \textbf{tabla de símbolos} con soporte de
          scopes anidados (global y local por función).
    \item Gestionar el \textbf{alcance y visibilidad} de variables,
          parámetros y funciones.
    \item \textbf{Verificar tipos} en expresiones, asignaciones,
          retornos y llamadas a funciones.
    \item \textbf{Detectar errores semánticos} y reportarlos con
          mensajes descriptivos.
    \item \textbf{Anotar el AST} con el tipo inferido de cada nodo.
    \item \textbf{Validar el significado del programa} detectando
          usos inválidos como operaciones aritméticas con booleanos
          o funciones sin instrucción \texttt{return}.
\end{enumerate}

% ──────────────────────────────────────────
\subsection{Tabla de símbolos}

La tabla de símbolos es una pila de diccionarios donde el nivel 0
corresponde al scope global y cada función abre un scope nuevo al
entrar y lo cierra al terminar su cuerpo. Cada entrada almacena
los siguientes atributos:

\begin{table}[H]
\centering
\renewcommand{\arraystretch}{1.2}
\begin{tabular}{@{}p{2.8cm} p{2.4cm} p{3.2cm} p{4.6cm}@{}}
\toprule
\textbf{Atributo} & \textbf{Tipo} & \textbf{Ejemplo} & \textbf{Descripción} \\
\midrule
\texttt{nombre}    & \texttt{str}  & \texttt{"suma"}
    & Identificador del símbolo \\
\texttt{tipo}      & \texttt{str}  & \texttt{"int"}
    & Tipo de dato (\texttt{None} para funciones) \\
\texttt{categoria} & \texttt{str}  & \texttt{"funcion"}
    & \texttt{variable}, \texttt{parametro} o \texttt{funcion} \\
\texttt{params}    & \texttt{list} & \texttt{['int','int']}
    & Tipos de parámetros (solo funciones) \\
\texttt{retorno}   & \texttt{str}  & \texttt{"int"}
    & Tipo de retorno (solo funciones) \\
\bottomrule
\end{tabular}
\caption{Atributos de una entrada en la tabla de símbolos}
\end{table}

Las operaciones de la tabla son:

\begin{itemize}
    \item \texttt{insertar(simbolo)}: agrega el símbolo al scope
          actual; devuelve \texttt{False} si ya existe en ese scope
          (redeclaración).
    \item \texttt{buscar(nombre)}: recorre los scopes de adentro
          hacia afuera y devuelve el primer símbolo encontrado
          o \texttt{None}.
    \item \texttt{entrar\_scope()} / \texttt{salir\_scope()}:
          gestión de la pila al entrar y salir de funciones.
\end{itemize}

% ──────────────────────────────────────────
\subsection{Manejo de alcance y visibilidad}

Cada función abre un scope local al momento de ser analizada.
Los parámetros se insertan en ese scope local, por lo que no son
visibles fuera de la función. Las variables globales sí son
accesibles desde cualquier scope interior gracias al recorrido
de adentro hacia afuera en \texttt{buscar()}. Al terminar el
análisis del cuerpo de la función, el scope local se descarta
con \texttt{salir\_scope()}.

% ──────────────────────────────────────────
\subsection{Verificación de tipos}

\subsubsection{Compatibilidad en asignaciones}

Una expresión de tipo $T_e$ puede asignarse a una variable de
tipo $T_v$ si se cumple:

\[
T_e = T_v \quad \text{o} \quad
(T_v = \texttt{float} \;\land\; T_e = \texttt{int})
\]

Esto permite promover enteros a flotantes, pero no al contrario.
Asignar \texttt{float} a \texttt{int} es un error semántico.

\subsubsection{Tipo resultante de operaciones aritméticas}

Los operadores relacionales (\texttt{==}, \texttt{!=}, \texttt{<},
\texttt{>}) siempre producen \texttt{bool} y requieren operandos
numéricos. Los operadores aritméticos (\texttt{+}, \texttt{-},
\texttt{*}, \texttt{/}) rechazan operandos de tipo \texttt{bool}.

\subsection{Anotación del AST}

Durante el recorrido del árbol, cada nodo recibe el atributo
\texttt{tipo\_anotado} con el tipo inferido. Esto enriquece el
AST con información semántica que puede ser aprovechada por fases
posteriores como la generación de código intermedio. Los nodos
que reciben anotación son: \texttt{NodoDeclaracion},
\texttt{NodoAsignacion}, \texttt{NodoBinario}, \texttt{NodoID},
\texttt{NodoNumero}, \texttt{NodoBooleano} y \texttt{NodoLlamada}.

% ──────────────────────────────────────────
\subsection{Implementación}

El módulo \texttt{semantico.py} importa directamente las clases
del lexer y del parser. Utiliza el patrón \textit{visitor}
implementado con \texttt{isinstance()} para despachar cada tipo
de nodo a su método correspondiente. El flujo de ejecución es:

\begin{enumerate}
    \item Leer y tokenizar el archivo con la clase \texttt{Lexer}.
    \item Construir el AST con \texttt{Parser.parsear()}.
    \item Recorrer el AST con \texttt{AnalizadorSemantico.analizar()}.
    \item Imprimir la tabla de símbolos y el AST anotado.
    \item Reportar los errores encontrados o confirmar que el
          programa es semánticamente correcto.
\end{enumerate}

Se ejecuta desde la terminal con:

\begin{lstlisting}[language=bash, caption={Ejecución del analizador semántico}]
python semantico.py archivo.peka
\end{lstlisting}

A continuación se muestra el visitante para declaraciones de
variables, que ilustra la integración entre tabla de símbolos,
verificación de tipos y anotación del AST:

\begin{lstlisting}[language=Python,
    caption={Visitante para declaración de variable}]
def _v_Declaracion(self, nodo):
    tipo      = nodo.tipo
    nombre    = nodo.nombre
    tipo_expr = self._visitar(nodo.valor)

    if tipo_expr is not None and not asignable(tipo, tipo_expr):
        self._error(
            f"No se puede asignar tipo '{tipo_expr}' a variable "
            f"'{nombre}' de tipo '{tipo}'"
        )

    nodo.tipo_anotado = tipo

    sim = Simbolo(nombre, tipo, 'variable')
    if not self.tabla.insertar(sim):
        self._error(
            f"Variable '{nombre}' ya fue declarada en este scope"
        )
    return tipo
\end{lstlisting}

\section{Generación de código intermedio y optimización}

El módulo \texttt{generador\_tac.py} recibe el AST validado
semánticamente y produce código de tres direcciones (TAC).
Cada instrucción tiene como máximo tres operandos:
\texttt{dest = izq op der}. Tras la generación se aplican
dos optimizaciones y una validación final.

% ──────────────────────────────────────────
\subsection{Instrucciones TAC generadas}

\begin{table}[H]
\centering
\renewcommand{\arraystretch}{1.2}
\begin{tabular}{@{}p{3.8cm} p{4.2cm} p{4.8cm}@{}}
\toprule
\textbf{Instrucción} & \textbf{Forma} & \textbf{Uso} \\
\midrule
Asignación          & \texttt{x = y}            & Declaración y asignación \\
Binaria             & \texttt{t = a op b}       & Expresiones aritméticas y relacionales \\
Etiqueta            & \texttt{L:}               & Punto de salto \\
Salto condicional   & \texttt{if not c goto L}  & Control de flujo \\
Salto incondicional & \texttt{goto L}           & Fin de bloque if/while \\
Parámetro           & \texttt{param x}          & Argumento de función \\
Llamada             & \texttt{t = call f, n}    & Invocación de función \\
Retorno             & \texttt{return x}         & Valor de retorno \\
Imprimir            & \texttt{print x}          & Salida en consola \\
\bottomrule
\end{tabular}
\caption{Conjunto de instrucciones TAC de MiniPeka}
\end{table}

% ──────────────────────────────────────────
\subsection{Optimizaciones aplicadas}

Se aplican dos pasadas secuenciales sobre el TAC generado:

\begin{enumerate}
    \item \textbf{Propagación de constantes:} si una variable
          siempre contiene un literal conocido, sus usos se
          sustituyen por ese valor. Si ambos operandos de una
          operación son constantes, la expresión se evalúa en
          tiempo de compilación.

    \item \textbf{Eliminación de código muerto:} instrucciones
          que asignan a temporales (\texttt{t0}, \texttt{t1}$\ldots$)
          cuyos resultados nunca son leídos posteriormente
          se descartan del TAC final.
\end{enumerate}

% ──────────────────────────────────────────
\subsection{Validación del código generado}

Tras las optimizaciones se realiza una pasada de validación que
verifica:

\begin{itemize}
    \item Toda variable o temporal usada fue definida previamente.
    \item Todo salto apunta a una etiqueta existente en el TAC.
\end{itemize}

Si se detecta algún problema se reporta con el número de
instrucción correspondiente.

\subsection{Flujo de integración}

El generador ejecuta las cinco fases en cadena:

\begin{lstlisting}[language=bash,
    caption={Ejecución del generador TAC}]

\end{lstlisting}

\noindent
\texttt{Lexer} $\to$ \texttt{Parser} $\to$
\texttt{AnalizadorSemantico} $\to$ \texttt{GeneradorTAC}
$\to$ \texttt{propagar\_constantes} $\to$

\section{Alcance del proyecto}

\subsection{Lo que el proyecto SÍ incluye}

\begin{itemize}
    \item Analizador léxico completo con manejo de errores léxicos y
          reporte de línea y columna del error.

    \item Analizador sintáctico basado en gramática LL con manejo de
          errores sintácticos y mensajes descriptivos.

    \item Generación del árbol de sintaxis abstracta (AST) para cualquier
          programa válido en MiniPeka.

    \item Tabla de símbolos con soporte de scopes anidados (global y local
          por función).

    \item Análisis semántico: verificación de tipos, variables declaradas
          antes de su uso y compatibilidad de tipos en asignaciones.

    \item Generación de código de tres direcciones (TAC) para expresiones
          aritméticas, asignaciones y estructuras de control.

    \item Al menos una optimización básica: propagación de constantes.

    \item Pruebas de funcionamiento con al menos cinco ejemplos válidos
          y cinco inválidos.

    \item Documentación técnica completa de cada fase del compilador.

    \item Video explicativo de 5 minutos demostrando el funcionamiento.
\end{itemize}

\subsection{Lo que el proyecto NO incluye}

\begin{itemize}
    \item Generación de código ejecutable (código máquina o ensamblador).

    \item Tipos de datos compuestos como arreglos, structs o cadenas de
          texto.

    \item Recursión directa o indirecta entre funciones.

    \item Forma SSA (considerada opcional según el PIA).

    \item Manejo de excepciones o errores en tiempo de ejecución.

    \item Interfaz gráfica de usuario.

    \item Operadores lógicos (\texttt{and}, \texttt{or}, \texttt{not}).
\end{itemize}
